## Context

Holy Sheet 是一个全新的开源 Web 电子表格库项目。当前市场上的解决方案如 Handsontable 体积较大且非完全开源，Luckysheet/Univer 功能丰富但复杂度高。我们的目标是创建一个轻量级（< 60KB gzipped）、高性能（百万级单元格）、可扩展（插件系统）的 Canvas 表格引擎。

**技术约束**：

- 框架无关：核心库不依赖 React/Vue 等 UI 框架
- 零运行时依赖：核心包不引入第三方运行时依赖
- TypeScript：类型安全，完整的类型定义
- Canvas 2D：跨浏览器兼容，避免 WebGL 复杂性

## Goals / Non-Goals

**Goals:**

- 核心包体积 < 60KB gzipped，按模块拆分
- 支持 100 万单元格渲染，首屏 < 50ms，滚动 60fps
- 内存占用 100 万单元格 < 200MB（稀疏存储 + 样式共享）
- 完整的撤销/重做支持，命令模式架构
- 插件系统支持功能扩展（公式、剪贴板、Excel 导入导出）
- 与 Excel 数据模型兼容（不含图表）

**Non-Goals:**

- 不实现图表功能（可通过插件扩展）
- 不实现协同编辑（为未来预留接口但不在 1.0 范围内）
- 不提供 React/Vue/Svelte 封装组件（属于后续 Phase 8）
- 不实现打印功能

## Decisions

### 1. 数据存储：稀疏矩阵 vs 密集数组

**决策**：采用稀疏矩阵 `Record<number, Record<number, CellData>>`

**理由**：

- 典型表格数据稀疏度高（大量空单元格）
- 与 Univer 数据结构保持一致，便于参考
- JSON 序列化友好，无需自定义序列化器
- 内存效率高，只存储有数据的单元格

**备选方案**：

- Map<string, CellData> 以 "row,col" 为 key：序列化不便
- 二维数组 CellData[][]：空间浪费，不适合大表格

### 2. 渲染架构：分层 Canvas vs 单一 Canvas

**决策**：三层架构（Main Canvas + Selection Canvas + DOM Layer）

**理由**：

- Selection Layer 需要频繁重绘（鼠标移动），独立 Canvas 避免全量重绘
- Main Layer 使用脏矩形增量渲染，提高性能
- DOM Layer 处理输入框和滚动，利用浏览器原生能力

**备选方案**：

- 单一 Canvas：简单但性能差，选区变化触发全量重绘
- 全 DOM 渲染：大数据量性能瓶颈明显

### 3. 坐标转换：分块累积缓存 vs 线性查找

**决策**：分块累积缓存（AccumulatorCache），块大小 1000 行

**理由**：

- 像素 → 行号转换是热点操作（滚动、点击）
- 分块缓存复杂度 O(log b + 1000)，优于 O(n)
- 支持局部失效，行高变化只需增量重建

**备选方案**：

- 完全缓存每行偏移：内存占用过大
- 线性查找：百万行性能不可接受

### 4. 索引映射：分段映射 vs 重建数据

**决策**：分段映射（IndexMapper + Segment）

**理由**：

- 插入/删除行列不移动实际数据
- 只更新映射表，O(k) 复杂度（k = 段数量）
- 适合频繁结构修改的场景

**备选方案**：

- 每次操作重建数据：大表格性能灾难
- 双向链表：实现复杂，遍历效率低

### 5. 样式存储：共享样式池 vs 内联样式

**决策**：StylePool 共享样式 + 支持内联样式

**理由**：

- 表格中大量单元格使用相同样式
- 样式去重显著减少内存占用
- 同时保留内联能力，支持特殊场景

**备选方案**：

- 纯内联样式：内存占用高
- CSS 类名：无法精确控制单元格样式

### 6. 命令系统：命令模式 vs 直接修改

**决策**：命令模式 + UndoStack + Transaction

**理由**：

- 统一的修改入口，便于追踪和审计
- 自然支持撤销/重做
- 事务支持批量操作原子性
- 为协同编辑预留扩展点

**命令合并策略**：300ms 内同一单元格连续输入合并为一个命令

### 7. 插件系统：钩子机制 vs 事件机制

**决策**：同步钩子机制

**理由**：

- 钩子可拦截和修改数据流
- 同步调用避免异步复杂性
- 按注册顺序执行，行为可预测

**钩子类型**：

- 生命周期钩子：onInstall, onUninstall
- 数据钩子：onCellGet, onCellSet
- 命令钩子：onBeforeCommand, onAfterCommand
- 渲染钩子：onCellRender
- 交互钩子：onKeyDown, onCellClick 等

## Risks / Trade-offs

### 性能风险

**[风险] Canvas 渲染瓶颈** → 采用多层优化：Path2D 批量绘制、脏矩形重绘、分层 Canvas、Worker 计算

**[风险] 内存占用过高** → 稀疏存储、StylePool 共享、视口外数据可选卸载

### 兼容性风险

**[风险] 浏览器 Canvas API 差异** → 特性检测、回退策略，优先支持 Chrome/Firefox/Safari 最新两个版本

**[风险] 触摸设备体验差** → 专门的 TouchHandler，支持点击/双击/长按/滑动/缩放

### 复杂度风险

**[风险] 公式引擎复杂度高** → 渐进实现，优先常用函数；作为独立包可选加载

**[风险] 包体积超标** → Tree-shaking 友好设计、代码审查、CI 体积检查

### 设计权衡

- **性能 vs 简单性**：选择复杂的分层渲染和缓存系统，换取大数据量下的流畅体验
- **体积 vs 功能**：核心包只含基础能力，公式/剪贴板/Excel 作为独立插件
- **灵活性 vs 一致性**：同步钩子限制了异步扩展，但保证了执行顺序可预测
